# 当前第一阶段实验逻辑说明

本文档整理当前项目中第一阶段最小实验流程的完整逻辑。当前阶段只包含模拟数据、样本构造、时间切分、Markov baseline 和评估，不包含 LSTM。

## 1. 实验目标

当前实验分两类预测任务。

### 1.1 应用间预测

输入：

- 当前时间特征
- 用户类型
- 历史前台应用序列
- 当前已经打开的应用集合

输出：

- 未来 3 分钟内可能使用的应用集合
- 未来 5 分钟内可能使用的应用集合
- 未来 10 分钟内可能使用的应用集合

这是多标签预测任务。一个未来窗口内可能出现多个应用，所以标签是 multi-hot 向量。

### 1.2 应用内预测

输入：

- 当前应用
- 当前时间特征
- 用户类型
- 当前应用内历史操作序列

输出：

- 下一步应用内操作

这是单标签预测任务。每个样本只预测一个 `label_next_op`。

## 2. 模拟原始日志生成逻辑

脚本：

```text
scripts/make_synthetic_data.py
```

输出：

```text
data/raw/app_events.csv
data/raw/op_events.csv
```

### 2.1 应用词表和操作词表

模拟数据读取：

```text
data/vocab/app_vocab.json
data/vocab/op_vocab.json
data/vocab/user_group_vocab.json
```

所有生成出来的应用名必须来自 `app_vocab.json`。

应用内操作必须来自 `op_vocab.json[app]`，并且不会把 `<PAD>` 和 `<UNK>` 当作真实操作主动生成。

### 2.2 用户和时间

默认生成：

- 10 个用户
- 每个用户约 2 小时行为
- 应用切换间隔随机为 1 到 5 分钟
- 应用内操作间隔随机为 5 到 60 秒
- 每个应用会话内生成 3 到 10 个操作

随机种子固定，默认是：

```text
seed = 42
```

因此同样参数下可以复现同一批模拟数据。

### 2.3 用户类型偏好

不同用户类型有不同应用偏好。

例如：

- `办公人群` 偏向 WPS、飞书、腾讯QQ、华为浏览器、百度网盘
- `游戏用户` 偏向米哈游、逆水寒、明日方舟、燕云十六声、青云诀
- `影音娱乐用户` 偏向抖音、哔哩哔哩、腾讯视频、爱奇艺、优酷视频、斗鱼、虎牙直播
- `金融用户` 偏向同花顺、华为钱包、腾讯QQ、华为浏览器
- `通用用户` 随机混合

偏好通过加权采样实现，不是硬规则。

### 2.4 `opened_apps` 的状态逻辑

当前模拟逻辑中，`opened_apps` 表示“电脑当前已经打开且尚未关闭的应用集合”。

每个用户开始时：

```text
opened_apps = 空
```

然后按时间推进：

1. 第一条应用事件会打开一个前台应用。
2. 如果下一次切换到已经打开的应用，`opened_apps` 不新增应用。
3. 如果下一次切换到新应用，`opened_apps` 最多新增一个应用。
4. 前台应用一定包含在 `opened_apps` 中。
5. 系统会以一定概率关闭一个非前台应用。
6. 关闭时一次最多关闭一个应用。
7. 默认最多同时保留 6 个打开应用，可通过 `--max-opened-apps` 调整。

因此 `opened_apps` 不会从 1 个应用突然随机跳到 5 个应用。它是由“打开一个新应用”和“关闭一个旧应用”的生命周期状态逐步演化出来的。

### 3.2 历史窗口

默认：

```text
history_len = 5
```

对每个时间点 `t`，取过去 5 个 `foreground_app` 作为：

```text
history_apps
```

`history_apps` 中保存的是应用 ID，不是应用名。

示例：

```text
米哈游 -> 飞书 -> 腾讯视频 -> 像素蛋糕 -> 腾讯视频
```

会映射成：

```text
[20, 1, 10, 29, 10]
```

具体 ID 由 `app_vocab.json` 决定。

### 3.3 当前打开应用

当前行的 `opened_apps` 会被转成 multi-hot 向量。

如果词表里有 30 个应用，则 `opened_apps` 向量长度就是 30。

某个位置为 1 表示该应用当前打开；为 0 表示当前未打开。

### 3.4 时间特征

每个样本包含：

```text
time_feature = [hour, weekday, is_weekend]
```

含义：

- `hour`：0 到 23
- `weekday`：0 到 6，周一为 0
- `is_weekend`：是否周末，0 或 1

### 3.5 用户类型

`user_group` 会映射成 ID。

如果原始日志里的用户类型不在 `user_group_vocab.json` 中，则回退为：

```text
通用用户
```

### 3.6 未来窗口标签

当前应用间任务有三个预测窗口：

```text
3min
5min
10min
```

对每个样本时间点 `t`：

- `label_3min`：未来 3 分钟内出现过的 `foreground_app`
- `label_5min`：未来 5 分钟内出现过的 `foreground_app`
- `label_10min`：未来 10 分钟内出现过的 `foreground_app`

标签也是 multi-hot 向量。

如果三个未来窗口里都没有任何应用，则跳过该样本。

## 4. 应用内样本构造逻辑

脚本：

```text
src/data/build_op_dataset.py
```

输入：

```text
data/raw/op_events.csv
data/vocab/op_vocab.json
data/vocab/user_group_vocab.json
```

输出：

```text
data/processed/op_samples.pkl
```

### 4.1 排序和分组

处理流程：

1. 读取 `op_events.csv`。
2. 跳过不在 `op_vocab.json` 中的应用事件。
3. 按 `user_id + app` 分组。
4. 每个分组内部按 `timestamp` 升序排序。

这样做的含义是：不同用户、不同应用的操作序列彼此独立。

### 4.2 历史操作窗口

默认：

```text
history_len = 4
```

对每个应用内操作时间点，取过去 4 个操作作为：

```text
history_ops
```

然后把当前操作作为：

```text
label_next_op
```

### 4.3 操作映射

操作 ID 来自：

```text
op_vocab.json[app]
```

规则：

- 不把 `<PAD>` 当作真实操作。
- 未知操作映射为 `<UNK>`。
- `history_ops` 里可以出现 `<UNK>`。
- `label_next_op` 不使用 `<PAD>`。

## 6. 应用间 Markov baseline 逻辑

模型：

```text
src/models/app_markov.py
```

训练评估：

```text
src/train/train_app_markov.py
```

### 6.1 当前模型形式

当前应用间 Markov 是 horizon-specific 的。

也就是说，3 分钟、5 分钟、10 分钟分别维护转移表：

```text
3min:  context -> app
5min:  context -> app
10min: context -> app
```

代码中的结构等价于：

```python
self.transitions[horizon][context][app] += 1
```

这样模型可以分别学习：

- 短期内常出现哪些应用
- 中期内常出现哪些应用
- 长期内常出现哪些应用

### 6.2 训练时统计什么

对于一个应用间样本：

```text
history_apps = [A, B, C, D, E]
```

如果是四阶 Markov：

```text
context = [B, C, D, E]
```

训练时只统计：

```text
[B, C, D, E] -> label_3min 中的应用
[B, C, D, E] -> label_5min 中的应用
[B, C, D, E] -> label_10min 中的应用
```

如果是四阶模型，还会同时保存三阶、二阶和一阶退避上下文：

```text
[C, D, E] -> label_xmin 中的应用
[D, E] -> label_xmin 中的应用
[E] -> label_xmin 中的应用
```

### 6.3 当前不统计什么

当前模型不统计 `history_apps` 内部转移。

也就是说，对于：

```text
A -> B -> C -> D -> E
```

不会额外统计：

```text
A, B -> C
B, C -> D
C, D -> E
```

原因是当前任务不是预测历史序列内部的下一个应用，而是预测：

```text
最近历史应用 -> 未来 3/5/10 分钟应用集合
```

这也避免了滑动窗口样本造成历史内部转移被重复放大的问题。

### 6.4 全局退避统计

当前全局频率只统计训练集中的未来标签。

也就是说：

```text
global_counts[horizon][app]
```

只来自：

```text
label_3min / label_5min / label_10min
```

不会把 `history_apps` 中出现过的应用混进去。

### 6.5 预测时退避顺序

如果当前是四阶模型，预测时退避顺序为：

```text
四阶 context
  -> 三阶 context
  -> 二阶 context
  -> 一阶 context
  -> 当前 horizon 的全局未来标签频率
```

例如：

```text
history_apps = [A, B, C, D, E]
order = 4
horizon = 10
```

预测时先查：

```text
[B, C, D, E]
```

如果没见过，依次再查：

```text
[C, D, E]
[D, E]
[E]
```

如果还没见过，再使用：

```text
global_counts[10]
```

### 6.6 评估指标

应用间预测是多标签任务，使用：

- Hit@K
- Recall@K
- Precision@K
- MRR

输出：

```text
outputs/results/app_markov_results.csv
```

CSV 字段：

```text
model,order,horizon,k,hit_at_k,recall_at_k,precision_at_k,mrr,num_samples
```

## 7. 应用内 Markov baseline 逻辑

模型：

```text
src/models/op_markov.py
```

训练评估：

```text
src/train/train_op_markov.py
```

### 7.1 当前模型形式

应用内 Markov 是按应用分别建模的。

结构可以理解为：

```text
app -> context -> next_operation
```

每个应用有自己的操作转移统计。

### 7.2 训练时统计什么

对于一个样本：

```text
app = 抖音
history_ops = [打开抖音, 浏览首页推荐, 点击视频, 播放视频]
label_next_op = 点赞视频
```

如果是四阶模型，则统计：

```text
[打开抖音, 浏览首页推荐, 点击视频, 播放视频] -> 点赞视频
```

模型也会从历史操作序列内部统计同一个应用下的操作转移，用于学习操作序列规律。

### 7.3 预测时退避

当前应用内模型如果高阶上下文没见过，会逐级退回低阶上下文。

也就是说：

```text
四阶 context
  -> 三阶 context
  -> 二阶 context
  -> 一阶 context
  -> 当前 app 的全局操作频率
```

### 7.4 特殊 token 处理

预测时：

- 不会把 `<PAD>` 作为预测结果。
- 尽量不把 `<UNK>` 排在前面，除非没有其他候选。

### 7.5 评估指标

应用内预测是单标签任务，使用：

- Top-K Accuracy
- MRR

输出：

```text
outputs/results/op_markov_results.csv
```

CSV 字段：

```text
app,model,order,k,topk_accuracy,mrr,num_samples
```

结果中每个应用会有单独行，同时会额外输出：

```text
app = ALL
```

表示整体平均结果。
