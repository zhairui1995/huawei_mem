# MGLRU lru_gen_pages 接口说明

本文档说明实验内核新增的 debugfs 接口：

```text
/sys/kernel/debug/lru_gen_pages
```

该接口用于对指定 PFN 所属 folio 进行页级 MGLRU 状态查询、generation 移动和回收策略控制。

## 1. 使用前提

确认当前运行的是包含本实验修改的内核：

```bash
uname -r
```

输出应包含：

```text
-mglru
```

确认 debugfs 已挂载：

```bash
mount | grep debugfs || sudo mount -t debugfs debugfs /sys/kernel/debug
```

确认接口存在：

```bash
sudo ls -l /sys/kernel/debug/lru_gen_pages
sudo cat /sys/kernel/debug/lru_gen_pages
```

该接口通常需要 root 权限读写。

## 2. 接口命令总览

读取接口会显示当前支持的命令：

```text
# commands:
#   query <pfn>
#   promote <pfn> <nr_pages>
#   depromote <pfn> <nr_pages>
#   protect add|del <pfn> <nr_pages>
#   scan skip add|del <prio_min> <prio_max> <pfn> <nr_pages>
#   scan only add|del <prio_min> <prio_max> <pfn> <nr_pages>
#   clear all
# pfn flags skip_prio_min skip_prio_max only_prio_min only_prio_max hits
```

命令分类：

- `query`：查询指定 PFN 所在 folio 的 MGLRU 状态。
- `promote`：一次性把指定 folio 移到最新 generation。
- `depromote`：一次性把指定 folio 移到对应类型的最老 generation。
- `protect`：添加或删除硬保护策略。
- `scan skip`：在指定 reclaim priority 范围内跳过目标 folio。
- `scan only`：在指定 reclaim priority 范围内只允许目标集合继续进入 scan/isolate。
- `clear all`：清空所有持久策略。

## 3. 参数格式

### 3.1 pfn

`pfn` 是物理页帧号。

支持十进制：

```text
2150528
```

支持带 `0x` 前缀的十六进制：

```text
0x20d080
```

不要使用裸十六进制：

```text
20d080
```

原因是内核解析使用 `kstrtoul(str, 0, value)`。带 `0x` 前缀时按十六进制解析，不带前缀时按十进制解析。

### 3.2 nr_pages

`nr_pages` 表示从 `pfn` 开始覆盖多少个 4 KiB 页。

单页验证建议：

```text
1
```

如果目标 PFN 属于大 folio，接口内部会归一化到 folio head PFN；但为了避免误覆盖，初始验证仍建议使用 `1`。

### 3.3 prio_min / prio_max

`prio_min` 和 `prio_max` 表示 `scan_control.priority` 的闭区间。

常用宽范围：

```text
0 12
```

约束：

- `prio_min >= 0`
- `prio_max <= DEF_PRIORITY`
- `prio_min <= prio_max`

## 4. query

格式：

```text
query <pfn>
```

示例：

```bash
echo "query 2150528" | sudo tee /sys/kernel/debug/lru_gen_pages
sudo cat /sys/kernel/debug/lru_gen_pages
```

作用：

- 根据 PFN 找到 online page。
- 转换为 folio。
- 输出该 folio 当前所属 MGLRU 状态。

输出示例：

```text
# query:
# input_pfn head_pfn nr_pages memcg_id nid type zone lru active gen seq max_seq min_seq_anon min_seq_file latest oldest
2150528 2150528 1 101 0 file 0 1 1 3 431 431 428 428 1 0
```

字段说明：

- `input_pfn`：用户输入的 PFN。
- `head_pfn`：该 PFN 所属 folio 的 head PFN。
- `nr_pages`：该 folio 覆盖的 4 KiB 页数。
- `memcg_id`：`mem_cgroup_id(memcg)`，用于对应 `/sys/kernel/debug/lru_gen` 里的 memcg id。
- `nid`：NUMA node id。
- `type`：页类型，`anon` 或 `file`。
- `zone`：内存 zone 编号。
- `lru`：folio 当前是否在 LRU 上，`1` 表示在 LRU 上。
- `active`：当前 generation 是否属于 active generation。
- `gen`：folio flags 中记录的 MGLRU 环形 generation 下标。
- `seq`：根据当前 `max_seq` 和 `gen` 反推得到的 generation seq。
- `max_seq`：当前 lruvec 最新 generation seq。
- `min_seq_anon`：当前 lruvec 中 anon 页的最老 generation seq。
- `min_seq_file`：当前 lruvec 中文件页的最老 generation seq。
- `latest`：`seq == max_seq` 时为 `1`。
- `oldest`：`seq == min_seq[type]` 时为 `1`。

异常输出：

```text
2150528 - - - - - - - - - - - - - error=ENOENT
```

含义：该 PFN 当前无法找到 online folio。常见原因是 PFN 输入错误、页已经释放、页不再 present、PFN 被迁移或目标进程映射已变化。

## 5. promote

格式：

```text
promote <pfn> <nr_pages>
```

示例：

```bash
echo "promote 2150528 1" | sudo tee /sys/kernel/debug/lru_gen_pages
```

作用：

- 对指定 PFN 范围逐页查找 folio。
- 将 folio 的 MGLRU generation 移到当前 lruvec 最新 generation。
- 目标 generation 为：

```c
lru_gen_from_seq(lruvec->lrugen.max_seq)
```

验证：

```bash
echo "query 2150528" | sudo tee /sys/kernel/debug/lru_gen_pages
sudo cat /sys/kernel/debug/lru_gen_pages
```

成功标准：

```text
latest == 1
```

等价判断：

```text
seq == max_seq
```

注意：

- `promote` 是一次性操作，不会在策略列表中留下持久项。
- 如果目标 folio 不在 LRU 上，操作会失败或无效果。
- 如果目标 PFN 已变化，可能查询不到或作用到新的 folio。

## 6. depromote

格式：

```text
depromote <pfn> <nr_pages>
```

示例：

```bash
echo "depromote 2150528 1" | sudo tee /sys/kernel/debug/lru_gen_pages
```

作用：

- 对指定 PFN 范围逐页查找 folio。
- 根据 folio 类型选择对应的最老 generation。
- 对 `anon` 页，目标 seq 是 `min_seq_anon`。
- 对 `file` 页，目标 seq 是 `min_seq_file`。

目标 generation 为：

```c
lru_gen_from_seq(lruvec->lrugen.min_seq[type])
```

验证：

```bash
echo "query 2150528" | sudo tee /sys/kernel/debug/lru_gen_pages
sudo cat /sys/kernel/debug/lru_gen_pages
```

成功标准：

```text
oldest == 1
```

等价判断：

```text
type == anon: seq == min_seq_anon
type == file: seq == min_seq_file
```

注意：

- `depromote` 会让目标 folio 更接近 reclaim 候选。
- 实验时建议选择可重新采样、可恢复的页。
- `depromote` 也是一次性操作，不会在策略列表中留下持久项。

## 7. protect

格式：

```text
protect add <pfn> <nr_pages>
protect del <pfn> <nr_pages>
```

示例：

```bash
echo "protect add 2150528 1" | sudo tee /sys/kernel/debug/lru_gen_pages
sudo cat /sys/kernel/debug/lru_gen_pages

echo "protect del 2150528 1" | sudo tee /sys/kernel/debug/lru_gen_pages
```

作用：

- `protect add`：为目标 folio 添加保护策略。
- `protect del`：删除目标 folio 的保护策略。

命中时行为：

- 在 MGLRU reclaim 路径中跳过目标 folio。
- 将目标 folio 重新热化到最新 generation。
- 对应策略行的 `hits` 加 1。

注意：

- 这不是 `mlock()`。
- 这不是 page pin。
- 它只影响本实验插桩覆盖到的 MGLRU reclaim 路径。
- 策略按 PFN/head PFN 记录；如果 PFN 被释放后复用，策略可能作用到新 folio。

## 8. scan skip

格式：

```text
scan skip add <prio_min> <prio_max> <pfn> <nr_pages>
scan skip del <prio_min> <prio_max> <pfn> <nr_pages>
```

示例：

```bash
echo "scan skip add 0 12 2150528 1" | sudo tee /sys/kernel/debug/lru_gen_pages
sudo cat /sys/kernel/debug/lru_gen_pages

echo "scan skip del 0 12 2150528 1" | sudo tee /sys/kernel/debug/lru_gen_pages
```

作用：

- 当 reclaim priority 落在 `[prio_min, prio_max]` 范围内时，跳过目标 folio。
- 命中后目标 folio 会被重新热化。
- `hits` 增加。

适用场景：

- 验证指定页是否在 reclaim scan 中被跳过。
- 观察压力场景下 `hits` 是否增长。

## 9. scan only

格式：

```text
scan only add <prio_min> <prio_max> <pfn> <nr_pages>
scan only del <prio_min> <prio_max> <pfn> <nr_pages>
```

示例：

```bash
echo "scan only add 0 12 2150528 1" | sudo tee /sys/kernel/debug/lru_gen_pages
sudo cat /sys/kernel/debug/lru_gen_pages

echo "scan only del 0 12 2150528 1" | sudo tee /sys/kernel/debug/lru_gen_pages
```

作用：

- 在指定 priority 范围内启用 only 模式。
- 命中 only 集合的 folio 允许继续进入 scan/isolate。
- 未命中 only 集合的 folio 会被跳过。

风险：

- `scan only` 的影响面比 `protect` 和 `scan skip` 更大。
- 如果 only 集合太小，可能显著改变系统 reclaim 行为。

建议：

- 先验证 `query`、`promote`、`depromote`。
- 再验证 `protect` 和 `scan skip`。
- 最后再测试 `scan only`。
- 测试后执行 `clear all`。

## 10. clear all

格式：

```text
clear all
```

示例：

```bash
echo "clear all" | sudo tee /sys/kernel/debug/lru_gen_pages
sudo cat /sys/kernel/debug/lru_gen_pages
```

作用：

- 删除所有 `protect`、`scan skip`、`scan only` 策略项。
- 不回滚已经执行过的 `promote` 或 `depromote`。

验证：

```bash
sudo cat /sys/kernel/debug/lru_gen_pages
```

期望策略列表为空，只剩命令帮助和表头。

## 11. 策略列表输出

读取接口时，持久策略会以如下格式输出：

```text
pfn flags skip_prio_min skip_prio_max only_prio_min only_prio_max hits
```

示例：

```text
2150528 PS- 0 12 0 12 3
```

字段说明：

- `pfn`：策略记录的 folio head PFN。
- `flags`：策略标志，三位字符。
- `P`：protect 生效。
- `S`：scan skip 生效。
- `O`：scan only 生效。
- `-`：对应策略未设置。
- `skip_prio_min`：skip 策略 priority 下界。
- `skip_prio_max`：skip 策略 priority 上界。
- `only_prio_min`：only 策略 priority 下界。
- `only_prio_max`：only 策略 priority 上界。
- `hits`：该策略在 reclaim 路径中命中的次数。

注意：如果 flags 中没有 `O`，即使 `only_prio_min/only_prio_max` 显示默认值，only 策略也不生效。skip 同理。

## 12. 推荐验证命令

以 PFN `2150528` 为例：

```bash
pfn=2150528
```

先清空旧策略：

```bash
echo "clear all" | sudo tee /sys/kernel/debug/lru_gen_pages
```

查询初始状态：

```bash
echo "query $pfn" | sudo tee /sys/kernel/debug/lru_gen_pages
sudo cat /sys/kernel/debug/lru_gen_pages
```

验证 promote：

```bash
echo "promote $pfn 1" | sudo tee /sys/kernel/debug/lru_gen_pages
echo "query $pfn" | sudo tee /sys/kernel/debug/lru_gen_pages
sudo cat /sys/kernel/debug/lru_gen_pages
```

检查：

```text
latest == 1
```

验证 depromote：

```bash
echo "depromote $pfn 1" | sudo tee /sys/kernel/debug/lru_gen_pages
echo "query $pfn" | sudo tee /sys/kernel/debug/lru_gen_pages
sudo cat /sys/kernel/debug/lru_gen_pages
```

检查：

```text
oldest == 1
```

验证 protect：

```bash
echo "protect add $pfn 1" | sudo tee /sys/kernel/debug/lru_gen_pages
sudo cat /sys/kernel/debug/lru_gen_pages
```

清理：

```bash
echo "clear all" | sudo tee /sys/kernel/debug/lru_gen_pages
```

## 13. 常见错误和处理

### 13.1 Permission denied

原因：没有 root 权限或 debugfs 权限不足。

处理：

```bash
sudo cat /sys/kernel/debug/lru_gen_pages
```

写入时也需要 sudo：

```bash
echo "query 2150528" | sudo tee /sys/kernel/debug/lru_gen_pages
```

### 13.2 No such file or directory

原因可能是：

- 没有进入实验内核。
- debugfs 未挂载。
- 内核没有包含本实验补丁。

处理：

```bash
uname -r
mount | grep debugfs || sudo mount -t debugfs debugfs /sys/kernel/debug
sudo ls /sys/kernel/debug/lru_gen_pages
```

### 13.3 query 输出 error=ENOENT

原因可能是：

- PFN 错误。
- PFN 使用了错误格式。
- 页已经不 present。
- 目标进程或映射已经变化。
- 页迁移或释放后 PFN 不再对应原页。

处理：

- 重新从 `page_data.tsv` 或当前进程 pagemap 采样 PFN。
- 十六进制 PFN 使用 `0x` 前缀。
- 优先选择仍 present 的匿名页或堆页。

### 13.4 latest/oldest 没有变化

原因可能是：

- 目标 folio 不在 LRU 上。
- PFN 已经变化。
- MGLRU seq 在查询期间推进，导致观察窗口变化。
- 命令作用到 folio head PFN，但用户对比了非 head PFN 的旧记录。

处理：

```bash
echo "query $pfn" | sudo tee /sys/kernel/debug/lru_gen_pages
sudo cat /sys/kernel/debug/lru_gen_pages
```

重点看：

```text
head_pfn nr_pages lru type seq max_seq min_seq_anon min_seq_file latest oldest
```
