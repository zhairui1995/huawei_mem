五、关键发现

发现 1：切后台触发主动 Swap
快照#1→#2（前台→后台），RSS 从 640MB 降至 560MB（-12.5%），Swap 从 55MB 飙升至 101MB（+83%）。 HarmonyOS 内核在应用进入后台时将约 46MB 的匿名页换出到磁盘，同时回收了部分文件映射页。
发现 2：ArkTS Heap 是最大且最敏感的内存消费者
"other" 类型（含 ArkTS Heap、partition_alloc）占 RSS 的 52-57%。前台→后台期间，other 类型 RSS 从 367MB 骤降至 290MB（-77MB），证明 ArkTS GC 在后台触发了主动释放。
发现 3：重启后 53MB 核心内存始终驻留
Fuzzy 模式下 775 个 VMA（15.5%）在 3 次快照中始终保持物理驻留，包括： ArkTS 运行时核心（huge object space、shared old space）、系统库（libutils、libhilog、libc++）、 以及 /dev/__parameters__/ 参数文件映射。这 53MB 是斗鱼的"基础内存足迹"。
发现 4：.so 共享库物理页在后-前台之间大面积进出
多个系统 .so 的物理页在后台被完全回收（present_ratio 100%→0%），重启后又重新加载（0%→100%）。 例如 libconfigpolicy_util.z.so、libdfx_dumpcatcher.z.so、libstack_printer.z.so 等， 说明 HarmonyOS 对后台进程的文件映射页采取了激进的回收策略。
发现 5：虚拟地址空间极大但利用极低（Chromium 特征）
三个快照的物理驻留率分别为 1.48%、1.17%、1.40%。~12M 虚拟页中仅 ~170K 实际分配物理内存， 剩余 98.5% 是 Chromium partition_alloc 预留的 4 个 ~16GB 虚拟地址池，不消耗物理内存。
发现 6：VMA 匹配率揭示 ASLR 效应
同 PID 快照间 VMA 匹配率 97.7%（2477/2519），仅 2.3% 的 VMA 新增或消失。 跨重启后匹配率降至 80.2%（2069/2580），约 20% 的 VMA 因 ASLR 地址漂移无法精确匹配， 验证了 fuzzy 匹配模式的必要性。

七、结论与后续

结论证据
HarmonyOS 对后台应用有主动内存回收后台 RSS -12.5%，Swap +83%，.so 物理页被大面积回收
斗鱼基础内存足迹约 53MB跨重启 fuzzy 模式 Hot RSS = 53.1MB
ArkTS 运行时是最大内存消费者other 类型占 RSS 52-57%，前台↔后台波动 77MB
ASLR 导致 ~20% VMA 地址漂移同PID匹配率97.7% vs 跨PID匹配率80.3%
Chromium partition_alloc 虚拟空间极大但不占物理内存物理驻留率仅 1.2-1.5%
